# mikata-web プロジェクト

WordPress 24サイトの Next.js App Router への移行プロジェクト

## 技術スタック

- **フレームワーク**: Next.js 15 (App Router)
- **言語**: TypeScript (strict mode)
- **スタイリング**: Tailwind CSS v4
- **パッケージ管理**: pnpm + Turborepo
- **デプロイ**: Vercel

## プロジェクト構成

```
mikata-web/
├── apps/
│   ├── bluecavetour/     # 石垣島 青の洞窟ツアーズ
│   ├── snorkeling/       # （次のサイト）
│   └── ...               # 計24サイト予定
├── packages/
│   ├── ui/               # 共通UIコンポーネント
│   ├── seo/              # SEO関連（メタタグ、JSON-LD）
│   └── config/           # 共通設定（eslint, tsconfig等）
├── turbo.json
└── pnpm-workspace.yaml
```

## 移行方針

### SEO要件（最重要）

- **既存URLパスを完全維持**（リダイレクトなしで同じURLで動作）
- meta情報（title, description）は既存サイトと同一
- OGP情報を維持
- JSON-LD構造化データを維持
- サイトマップ自動生成（/sitemap.xml）

### 共通コンポーネント化

24サイトで共通化できる要素：
- ヘッダー / フッター
- プラン一覧・詳細のレイアウト
- 予約フォーム
- SEO関連（generateMetadata, JSON-LD）
- 画像最適化（next/image wrapper）

```
│   └── lib/             # ユーティリティ
├── public/
└── next.config.ts
```

### コンポーネント

- 共通コンポーネントは `packages/ui` に配置
- サイト固有のコンポーネントは `apps/xxx/src/components` に配置
- Server Components をデフォルトとし、必要な場合のみ 'use client'

### スタイリング

- Tailwind CSS v4 のユーティリティクラスを使用
- サイトデザインは基本的に既存と同じにする
- 今後のメンテナンス性を考えて、タグ構造は整理してきれいにする
- タグ構造を整理した結果、よく見ないとわからない程度であれば見た目が変わることは許容する
- 見た目が変わる部分については常に理由を説明して変わっても良いか確認をしてください
- 複雑なスタイルは `@apply` ではなく、コンポーネント内で管理
- レスポンシブはモバイルファースト
- **重要: Tailwind CSS v4 の `@source` ディレクティブ**
  - v4ではクラスの自動検出がデフォルトで同一パッケージ内のみ
  - モノレポで `packages/ui` のTailwindクラスを使うには、各appの `globals.css` に `@source` を追加する必要がある
  ```css
  @source "../../../../packages/ui/src/**/*.tsx";
  ```
  - これがないと `hidden lg:flex` 等のレスポンシブクラスが生成されず、UIが崩れる

### 既存サイトとの見た目合わせ作業手順

既存サイトの見た目を再現する際は、以下の手順を必ず守ること:

1. **実装前に既存サイトの実値を調査する**
   - 色、サイズ、フォント、余白等のスタイル値は推測せず、既存サイトのDevTools（computed style）やソースコードから実際の値を取得してから実装する
   - SVGアイコンの場合はSVGソースを取得し、fill/stroke/色を確認する
   - 「おそらくこの色だろう」という推測での実装は禁止

2. **実装前に既存サイトのHTML構造（親子関係）を調査する**
   - 複数の要素が1つのカード/パネルにまとまっているか、別々のカード/パネルに分かれているかを必ず確認する
   - コンポーネントの分割方針は既存サイトの要素グループ構造に従う。独自判断で分割・統合しない
   - 背景色・ボーダー・border-radiusが適用される範囲（どの親要素に設定されているか）を調査し、その構造を忠実に再現する
   - 過去の失敗例: カテゴリタブと検索フォームが既存サイトでは1つの白カード内にあったが、新サイトでは別々のカード（別背景色）として実装してしまった

3. **実装後にブラウザで並べて比較する**
   - 既存サイトと新サイトのスクリーンショットを並べて、差異がないか目視確認する
   - コードからの予測（「このコードならこう表示されるはず」）は一切しない
   - 必ずスクリーンショットで実際の描画結果を確認する

4. **差異を見つけたら自発的に修正する**
   - ユーザーからの指摘を待たず、発見した差異は自分で修正する
   - 修正後に再度スクリーンショットで確認する

5. **「一致している」と報告する前の最終チェック**
   - スクリーンショット比較なしに「一致」と報告してはならない
   - 特に背景色・カード構造・区切り線・角丸など「レイアウト構造レベル」の差異を見逃さないこと

- `public/images/sidebar/` - サイドバーバナー（フェリー予約、プレミアムプラン）
